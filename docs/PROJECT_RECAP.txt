================================================================================
 ARCHITECTURE
================================================================================

  AUTH FLOW:
    1. landing.html → Supabase signInWithPassword / signUp
    2. On success → redirect to index.html
    3. sidebar.js loads on every page, checks auth, redirects to landing.html if no session
    4. sidebar.js creates window.db (Supabase client), stores session/profile on window
    5. sidebar.js registers onAuthStateChange to keep window.currentSession live
    6. ai-assistant-v4.js calls window.db.auth.refreshSession() before every request
       (KEY: refreshSession() guarantees a fresh JWT; getSession() was returning stale tokens)

  SIDEBAR (sidebar.js):
    - Self-executing IIFE injected into every page via <script src="sidebar.js">
    - Creates Supabase client (window.db)
    - Auth guard: redirects to landing.html if no session
    - Registers onAuthStateChange listener → updates window.currentSession on token refresh
    - Loads user profile from profiles table
    - Loads categories from table_definitions
    - Injects fixed 260px dark sidebar on desktop, hamburger menu on mobile
    - Applies theme from localStorage (inv_theme key)
    - Dispatches 'sidebar-ready' custom event when done

  THEME SYSTEM:
    - localStorage-first: theme stored in inv_theme key as JSON
    - Inline <script> in <head> applies CSS vars BEFORE render (prevents FOUC)
    - sidebar.js also applies theme as fallback
    - 20+ CSS custom properties for colors, sizing, spacing
    - Settings page has 6 presets + custom color pickers with live preview

  MULTI-USER / DATA ISOLATION (v4.0 — hardened):
    - Supabase Auth for real user accounts
    - profiles table auto-created via trigger on auth.users INSERT
    - table_definitions: user_id is NOT NULL (DB-level enforcement, no NULL rows possible)
      UNIQUE(table_name, user_id) — two users can each have their own "toners" table
      RLS: NULL-safe policies (auth.uid() IS NOT NULL AND user_id = auth.uid())
    - field_definitions: SELECT scoped via ownership JOIN to table_definitions
      INSERT/UPDATE/DELETE locked to service role only (edge function adminDb)
    - ui_settings: user_id NOT NULL, UNIQUE(key, user_id), NULL-safe RLS
    - Dynamic item tables: NULL-safe RLS on all four operations
    - Default UI settings auto-seeded per-user on signup via DB trigger
    - Edge function fire-and-forgets seed_default_ui_settings() on every request
      as a safety net for users created before the v4.0 migration
    - sanitizeTableName() in edge function prevents SQL injection in DDL

  EDGE FUNCTION (smart-api-v4.ts) — v4.0:
    - verify_jwt: false (gateway-level check disabled; see SUPABASE CONFIG note)
    - Token validation: adminDb.auth.getUser(token) → 401 if invalid/expired
    - Two Supabase clients:
        adminDb — service role key, DDL + field_definitions writes, bypasses RLS
        userDb  — user JWT in Authorization header, all DML, RLS scopes to auth.uid()
    - assertOwnership() helper: called before every item read/write to verify the
      requesting user owns the target table via RLS-scoped userDb query
    - sanitizeTableName(): strips non [a-z0-9_] chars from table names before DDL
    - DIRECT TOOL CALL MODE: { tool_call: { name, input } } bypasses AI loop
    - AI MODE: Claude tool-use loop (up to 8 iterations)
    - 12 AI tools (run_sql removed from AI; available via direct tool_call only):
        list_categories  — userDb, RLS auto-scopes to user
        get_items        — ownership check + userDb RLS
        get_fields       — ownership check + userDb RLS (field_definitions SELECT policy
                           also enforces ownership via JOIN)
        create_category  — userDb for table_definitions, adminDb for DDL + field_definitions
        rename_category  — userDb (RLS enforces ownership)
        delete_category  — ownership check via userDb, adminDb for DROP TABLE
        add_field        — ownership check via userDb, adminDb for ALTER TABLE
        upsert_item      — ownership check + userDb, explicit user_id stamp on INSERT
        delete_item      — ownership check + userDb RLS
        get_ui_settings  — userDb, RLS scopes to user
        set_ui_setting   — userDb, upserts with (key, user_id) composite key
        set_layout       — userDb, upserts with (key, user_id) composite key
        [run_sql — adminDb, available only via direct tool_call from trusted app code]

  AI ASSISTANT (ai-assistant-v4.js):
    - Floating chat widget (bottom-right corner, ✨ button)
    - Calls window.db.auth.refreshSession() before each message (guarantees fresh JWT)
    - Falls back to getSession() if refresh fails, then to cached window.currentSession
    - Shows clear "session expired" message if no token available (instead of cryptic error)
    - Sends page context (current page, table name) with every request
    - Triggers refresh events on data changes (ai-refresh, loadInventoryCards)

  EXPORT / IMPORT:
    - inventory.html: "⬇ Export CSV" downloads current table as Google Sheets-compatible CSV
        CSV format: #fields: mapping row, display name header, then data rows
        BOM included for correct UTF-8 in Excel / Google Sheets
    - inventory.html: "⬆ Import CSV" uploads a CSV, matches columns by field name or
        display name (case-insensitive), inserts rows with user_id, reports results
    - index.html: "⬇ Export All" loops through all user categories, downloads one CSV
        per category with 400ms delay between downloads

================================================================================
 DATABASE SCHEMA
================================================================================

  profiles:
    id (uuid PK, FK auth.users ON DELETE CASCADE)
    display_name, email, role, created_at
    RLS: SELECT/UPDATE/INSERT scoped to auth.uid() = id (NULL-safe)

  table_definitions:
    id (uuid PK), table_name (text NOT NULL), display_name, icon
    user_id (uuid NOT NULL, FK auth.users ON DELETE CASCADE)
    created_at
    UNIQUE (table_name, user_id)
    RLS: NULL-safe SELECT/INSERT/UPDATE/DELETE scoped to auth.uid() = user_id

  field_definitions:
    id (uuid PK), table_name (FK → table_definitions), field_name,
    display_name, field_type, required, sort_order, created_at
    UNIQUE (table_name, field_name)
    RLS SELECT: EXISTS (SELECT 1 FROM table_definitions WHERE table_name = ... AND user_id = auth.uid())
    RLS INSERT/UPDATE/DELETE: service role only (WITH CHECK (false) for user role)

  ui_settings:
    id (uuid PK), key (text NOT NULL), user_id (uuid NOT NULL, FK auth.users ON DELETE CASCADE)
    value, updated_at
    UNIQUE (key, user_id)
    RLS: NULL-safe SELECT/INSERT/UPDATE/DELETE scoped to auth.uid() = user_id

  Dynamic item tables (e.g. toners, monitors):
    id (uuid PK DEFAULT gen_random_uuid())
    user_id (uuid NOT NULL)  — no DEFAULT auth.uid() to avoid silent null inserts
    created_at (timestamptz DEFAULT now())
    ...custom fields (text / numeric / date)
    RLS: NULL-safe per-user policies on SELECT/INSERT/UPDATE/DELETE

  Helper functions:
    exec_sql(sql text) → jsonb
      SECURITY DEFINER, EXECUTE granted to service_role only
      Used by edge function adminDb for all DDL (CREATE/DROP/ALTER TABLE, policies)

    seed_default_ui_settings(target_user_id uuid) → void
      SECURITY DEFINER, EXECUTE granted to service_role only
      Inserts 26 default ui_settings rows for a user (ON CONFLICT DO NOTHING)
      Called by on_auth_user_created trigger and fire-and-forget in edge function

  Triggers:
    on_auth_user_created → handle_new_user()
      Fires AFTER INSERT ON auth.users
      Creates profile row + seeds default UI settings for every new signup

  Token / auth troubleshooting:
    - "Invalid JWT" errors: caused by getSession() returning a cached stale token.
      Fixed in v4.0: ai-assistant-v4.js uses refreshSession() instead.
    - If users see "session expired" messages, ensure sidebar.js onAuthStateChange
      listener is registering correctly and window.db is initialized before
      ai-assistant-v4.js runs.
    - verify_jwt: false is the correct setting (v39+). The function validates tokens
      manually via adminDb.auth.getUser(token). Setting verify_jwt: true caused
      gateway-level 401s for all requests regardless of token validity.
    - Flutter "AuthApiException: Invalid Refresh Token": user's refresh token was
      invalidated. Fixed by SessionExpiredException + _freshToken() in api_service.dart
      which signs out gracefully and navigates to LoginScreen.

  401 API error root cause + fix:
    - A previous AI (security AI) deployed edge function v35 with verify_jwt: true
      at the Supabase gateway level. For an unknown reason the Supabase gateway was
      rejecting valid user JWTs before the function code even ran (62–179ms response
      time vs 338–843ms for working requests). Pattern: v31/v33/v34 all 200s;
      v32/v35/v36/v37 all 401s.
    - Fix: deployed v38/v39 with verify_jwt: false. The function itself validates
      tokens via adminDb.auth.getUser(token) — no security loss, gateway just no
      longer blocks before function code runs. Confirmed fix by debug curl returning
      custom "Missing Authorization header" message (proving function code executing).

  Edge function — run_sql security decision:
    - run_sql removed from Claude's AI tools array (service role bypasses RLS,
      prompt injection risk). Comment in tools: "run_sql intentionally excluded —

      code only."

CHAT-UX UPGRADES (port concepts from the new app — small wins)
================================================================================

[3] STREAMDOWN — RICH MARKDOWN RENDERING WITH WIDGETS
    What it does: when AI emits a Markdown table or code block, browser renders it
    as a real <table>/<pre> with auto-attached buttons:
       - Tables  → Copy / Download CSV / View fullscreen
       - Code    → Copy / Run
       - Mermaid → renders as a real diagram
    Mechanism (plain English):
       - AI just emits plain Markdown (no change to backend / system prompt needed).
       - Browser-side library watches for Markdown patterns (| | |  ```  # etc.)
       - Builds proper HTML and wraps in a "card" with action buttons.
       - Re-parses on every streamed chunk so widgets fill in progressively.
    Effort: ~1 day total.
    Steps:
       a. Add Streamdown's vanilla-JS bundle via CDN <script> in HTML pages.
       b. In ai-assistant-v4.js, replace plain text rendering of assistant
          replies with: Streamdown.render(replyText, bubbleElement)
       c. Tweak system prompt in supabase/functions/smart-api/index.ts to encourage Markdown tables
          for list responses (e.g. "Show me all items" → table of items).

[4] STREAMING REPLIES (SSE) INSTEAD OF WAIT-THEN-DUMP
    Today: chat waits for the full Claude response, then drops it. ~3-5s spinner.
    Goal:  words appear as Claude types them. Same total time, feels much faster.
    Three layers all need fixing:
       a. supabase/functions/smart-api/index.ts:  add  stream: true  to the Anthropic fetch.
       b. supabase/functions/smart-api/index.ts:  return a ReadableStream that pipes Anthropic's SSE
                            chunks straight to the browser, instead of awaiting
                            resp.json() and returning a single JSON.
                            For tool-use loops: stream text deltas as they come,
                            pause stream when Claude wants a tool, run the tool,
                            resume the stream with the tool result.
       c. ai-assistant-v4.js:  replace  await response.json()  with
                            response.body.getReader() loop that appends each
                            chunk's text to the chat bubble (and feeds Streamdown
                            so widgets fill in live).
    Effort: ~half day if no tool-use, ~1 day with the tool-use loop.

[5] INLINE TOOL-CALL STATUS BADGES (the "Inventory updated" pills)
    What you saw on the new app: each time Claude calls a tool, a small green
    pill appears INSIDE the assistant bubble — "🔄 Inventory updated".
    Effect: user sees Claude's reasoning step by step instead of one big text
    blob at the end.
    Mechanism: when the SSE stream emits a TOOL_CALL_END event with a result,
    the chat UI inserts a small pill component above/below the next text chunk.
    Effort: ~half day on top of streaming (#4). Requires #4 first.

[6] AUTO-REFRESH OF VIEWS AFTER AI CHANGES DATA
    Already partly there: StockAI fires a `data-changed` CustomEvent.
    Improvement: also fire after EVERY tool-result event during a stream, so the
    inventory grid updates the instant the AI finishes adding/editing.
    Effort: ~1 hour once streaming is in.

LARGER QUESTION — MIGRATE OFF SUPABASE TO NETLIFY?
================================================================================

[7] SHOULD WE REBUILD STOCKAI ON NETLIFY'S STACK?
    Pros (vs current Supabase setup):
       - No 7-day pause (Netlify Database = Neon Postgres, data persists forever;
         5-min compute cold start instead).
       - One less account to manage (no Supabase).
       - Streaming chat is native via @tanstack/ai library.
       - AI Gateway pays the Anthropic bill until quota — but plan to plug in own
         key so you keep observability.
       - Modern stack (TanStack Start + React + Tailwind + Drizzle).
    Cons (what gets harder, not lost):
       - Less convenient DB admin (Supabase's table editor is slick; Neon's
         dashboard is bare-bones — would use DBeaver or netlify db CLI).
       - No Row-Level Security as a safety net — every read/write must be wrapped
         in a server function that enforces user_id ownership manually.
       - Frontend writes change shape: today browser calls db.from('toners').select()
         directly; on Netlify everything routes through ~10-14 server endpoints.
    What you do NOT lose if you port carefully:
       - Custom dynamic per-user tables (CREATE TABLE works on Neon too).
       - All 13 AI tools (port the implementations from Supabase calls to Drizzle).
       - Theme system (it's all browser-side, framework-agnostic).
       - CSV export (browser-side).
       - Flutter app (just point at new endpoints + swap auth client).
       - Multi-user isolation (enforce in code instead of RLS).
    Effort: ~7-10 working days for a careful port. Not 2-3 weeks.

       - Update Flutter app to point at new endpoints + swap auth:  ~2 days

    Result: two working versions. A/B test. Ship the winner, keep the other as backup.
